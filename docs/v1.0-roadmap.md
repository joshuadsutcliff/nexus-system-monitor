# Nexus System Monitor — v1.0 Roadmap

> **Approved:** 2026-03-24 · Based on v0.1.8.2 · **This is the authoritative planning document for all work from v0.2.0 through v1.0.0.**

---

## Definition of "1.0"

A user downloading v1.0 should be able to:

1. **Install without confusion** — clear install instructions, documented SmartScreen/Gatekeeper workarounds
2. **Use every visible feature** — nothing in the sidebar that silently fails or says "coming soon"
3. **Trust it with admin privileges** — test suite validates all core logic; no data-corruption bugs; app logs its actions
4. **Understand what they're looking at** — screenshots in README, empty/loading states, command palette for discoverability
5. **Know the project is alive** — current CHANGELOG, issue templates, update notifications
6. **Get more value over time** — health trends, resource predictions, and proactive warnings

Code signing deferred to post-1.0 (cost barrier for solo OSS — document workarounds clearly instead).

## Phase 20: Test Suite (→ v0.2.0)

**Why this first:** The app modifies process priorities, kills processes, changes CPU affinity, and manages services under admin privileges. There is 1 placeholder test. This is the single biggest credibility and safety gap.

**Scope:** 150–250 unit tests across 12–15 test files

### Infrastructure (build first)
- Mocks for `IProcessProvider`, `ISystemMetricsProvider`, `ILogger<T>`
- SQLite in-memory database helpers for store tests
- Rx test scheduler / fake observable helpers for service tests
- Start with pure-logic services that need none of the above

### Test File Priority Order
| Priority | File | Why |
|----------|------|-----|
| 1 | `RulesEngineTests` | Safety-critical — modifies process state |
| 2 | `ProcessPreferenceStoreTests` | Data persistence correctness |
| 3 | `MetricsStoreTests` | Storage integrity, rollup math |
| 4 | `SettingsServiceTests` | Config save/load, corruption recovery |
| 5 | `HealthScoringTests` + `ImpactScoreCalculatorTests` | Math validation |
| 6 | `SlidingStatsTests` | Statistical correctness |
| 7 | `AnomalyDetectionServiceTests` | Detection thresholds |
| 8 | `PerformanceProfileServiceTests` | Activation/deactivation lifecycle |
| 9 | `BottleneckDetectorTests` | Workload classification |
| 10 | `AlertsServiceTests` | Rule evaluation, event emission |
| 11 | `NmapXmlParserTests` | XML parsing edge cases |
| 12 | `PlatformCapabilitiesTests` | Mock provider validation |

### Key Files
- `tests/NexusMonitor.Core.Tests/PlaceholderTest.cs` — replace with real test suite
- `src/NexusMonitor.Core/Rules/RulesEngine.cs` — #1 test target
- `src/NexusMonitor.Core/Storage/MetricsStore.cs` — #3 test target
- `src/NexusMonitor.Core/Health/HealthScoring.cs` — #5 test target

### Risk
Rx mocking infrastructure (fake `Observable<ProcessInfo[]>`) takes more setup than writing the tests themselves. Build momentum first with pure-logic services.

## Phase 21: Bug Fixes + Dependency Stabilization (→ v0.3.0)

### Deferred Bug Fixes (D1–D11, prioritized)
| Priority | Bug | Fix |
|----------|-----|-----|
| 1 | **MetricsStore buffer growth** | Cap `FlushAll` buffer size or flush synchronously on overflow |
| 2 | **SQLite writer coordination** | Enforce single-writer discipline (WAL mode + app-level lock) |
| 3 | **CLI service lifecycle** | Wire `Console.CancelKeyPress` + `AppDomain.ProcessExit` to service teardown |
| 4 | **Linux Statvfs struct** | Verify ABI layout with `[StructLayout(Sequential)]` + correct field sizes |
| 5 | **COM reference leak** | `Marshal.ReleaseComObject` or `using` patterns in Windows provider |
| 6 | **Remaining empty catch blocks** | Replace with logged warnings |
| 7 | **GamingMode/ProBalance ViewModel Dispose** | Subscription cleanup on deactivate |

### Dependency Upgrades
- **LiveChartsCore** `2.0.0-rc4` → stable release (read release notes first — API may have breaking changes; test all chart views after upgrade)
- Audit all NuGet packages; pin every dependency to a stable, non-pre-release version

### Disk Analyzer: Ship It
The engine is fully built (MFT scanner, recursive scanner, treemap, duplicate finder). Work is testing + polish, not building.
- Remove "currently disabled" language from `README.md` and `feature-inventory.md`
- Test treemap rendering + MFT scan on Windows
- Verify graceful fallback to recursive scanner on macOS/Linux
- Update `gap-analysis.md` to reflect Disk Analyzer is now enabled

### Session Persistence (New Feature #9)
Remember across app restarts: window position/size, active tab, sidebar collapse state, DataGrid sort columns, scroll position.
- Extend `AppSettings` model with session state fields
- Extend `SettingsService` to save/restore those fields
- Save on `Window.Closing`, restore in `MainWindow.OnLoaded`
- **Also in this phase:** Close the Crystal Glass TODO at `MainWindow.axaml.cs:45` — current behavior (opt-in glass + Smart Tint) is the correct answer; document and remove the TODO

## Phase 22: New Features Batch 1 (→ v0.4.0)

### Command Palette (New Feature #4)
A searchable overlay (Ctrl+K) listing every available action: navigation, process commands, mode toggles.
- New `CommandPaletteView.axaml` — overlay panel with TextBox + filtered ListBox
- New `CommandPaletteViewModel.cs` — `ICommand` registry, fuzzy text match, keyboard nav (Up/Down/Enter/Escape)
- Register commands from existing ViewModels (navigate tabs, kill process, set priority, toggle modes)
- Wire `Ctrl+K` global keybinding in `MainWindow`

### Process Groups / Tags (New Feature #5)
User-defined process groupings with aggregate resource stats and rule targeting.
- New `ProcessGroup` model — name, color, list of process name patterns
- New `ProcessGroupStore` — SQLite table, thread-safe CRUD
- New `ProcessGroupsViewModel` + `ProcessGroupsView` for managing groups
- Extend `ProcessesViewModel` — group column, aggregate CPU/RAM per group as summary rows
- Extend `RulesEngine` — rules can target a group name instead of an individual process name

### Quiet Hours / Focus Mode (New Feature #10)
Scheduled low-activity windows where monitoring backs off and notifications suppress.
- New `QuietHoursService` in `Core/Automation` — time-window scheduler
- Extend `AppSettings` — quiet hours config (start time, end time, days of week, enabled toggle)
- Extend `SettingsView` — quiet hours configuration section
- When active: double all `Observable.Interval` periods; suppress `INotificationService` calls; show subtle status indicator in sidebar

### Key Files
- `src/NexusMonitor.UI/Views/CommandPaletteView.axaml` (new)
- `src/NexusMonitor.UI/ViewModels/CommandPaletteViewModel.cs` (new)

## Phase 23: New Features Batch 2 (→ v0.5.0)

### Webhook Notifications (New Feature #7)
External alert delivery via HTTP POST — unlocks Discord, Slack, email-via-Zapier, and server monitoring.
- New `WebhookNotificationChannel` in `Core/Alerts` — `HttpClient.PostAsync` JSON payload
- Extend `AppSettings` — webhook URL, optional auth header, enabled toggle
- Extend `AlertsService` — dispatch to webhook channels after in-app notification fires
- Payload: `{ "alert": "...", "severity": "warning", "timestamp": "...", "hostname": "...", "metrics": {...} }`
- Extend `SettingsView` — webhook config section with "Send Test" button
- Quiet Hours integration: suppress webhooks when quiet hours are active

### System Health Trends (New Feature #2)
Historical health score visualization — makes the Health Dashboard useful beyond right-now.
- Extend `MetricsStore` — add `health_scores` table or column to persist `SystemHealthService` score snapshots
- New `HealthTrendsViewModel` + section in `DashboardView` — sparkline chart (LiveCharts) showing score over days/weeks
- Summary text: "Avg health this week: 74 ↓ from 81. Top bottleneck: memory pressure"
- Leverage existing `BottleneckDetector` reports for trend attribution

### Resource Predictions (New Feature #1)
Proactive warnings powered by the existing `LinearRegression` class.
- New `PredictionService` in `Core/Health` — runs on a slow timer (every 5 min); queries `MetricsStore` for 24h/7d trends; uses `LinearRegression` to project disk depletion, RAM exhaustion
- Surface predictions in `DashboardView` as dismissable warning cards: "Disk C: full in ~14 days at current growth rate"
- Integrate with `AlertsService` — predictions can fire alerts (and therefore webhooks)
- Quiet Hours integration: suppress prediction alert firing during quiet hours

### Key Files
- `src/NexusMonitor.Core/Alerts/WebhookNotificationChannel.cs` (new)
- `src/NexusMonitor.Core/Health/PredictionService.cs` (new)
- `src/NexusMonitor.Core/Alerts/AlertsService.cs` — webhook dispatch
- `src/NexusMonitor.Core/Storage/MetricsStore.cs` — health_scores persistence
- `src/NexusMonitor.Core/Health/LinearRegression.cs` — reuse for predictions

## Phase 24: New Features Batch 3 (→ v0.6.0)

### Startup Impact Analysis (New Feature #3)
Measured, data-driven boot resource attribution (not Microsoft's opaque heuristic).
- New `StartupImpactService` in `Core` — on app launch, if system uptime < 3 min, record per-process CPU/RAM for 60–90s; attribute to startup items by matching process names to `IStartupProvider` entries
- Extend `StartupItem` model — `MeasuredImpact` field (CPU seconds, peak RAM MB)
- Persist measurements in SQLite (per-boot snapshots with timestamp)
- Extend `StartupViewModel` + `StartupView` — "Impact" column with mini bar chart, sortable

### Snapshot & Compare (New Feature #6)
Point-in-time system state diffing — a debugging superpower no consumer tool offers.
- New `SystemSnapshot` model — running processes (name, CPU, RAM), services (name, status), startup items (name, enabled)
- New `SnapshotService` in `Core` — `TakeSnapshot()`, `CompareSnapshots(a, b)` returns diff (added/removed/changed per category)
- Persist snapshots in SQLite with timestamp + optional user label
- New `SnapshotView.axaml` + `SnapshotViewModel` — "Take Snapshot" button, snapshot list, color-coded diff table (green = added, red = removed, yellow = changed)
- New sidebar nav entry

### Remote Instance Viewer (New Feature #8)
Read-only view of another Nexus instance — turns a single-PC tool into a network monitor.
- Verify `PrometheusExporter` exports all key metrics (CPU, RAM, disk, health score, process count)
- New `RemoteMonitorService` in `Core` — polls a configurable URL (another Nexus's Prometheus endpoint), parses text format into `SystemMetrics`-compatible model
- New `RemoteView.axaml` + `RemoteViewModel` — connection dialog (URL + optional label), read-only dashboard showing remote metrics in same layout as local Dashboard
- Extend `SettingsView` — saved remote connections list
- New sidebar nav entry

### Key Files
- `src/NexusMonitor.Core/StartupImpactService.cs` (new)
- `src/NexusMonitor.Core/Storage/SnapshotService.cs` (new)
- `src/NexusMonitor.Core/Models/SystemSnapshot.cs` (new)

## Explicitly Deferred to Post-1.0

| Feature | Reason |
|---------|--------|
| Code signing | Cost barrier (~$300/yr combined); document workarounds for 1.0 |
| Auto-update installer | Update *checker* ships in v0.9.0; auto-*update* is post-1.0 |
| Multi-select process actions | Significant UI complexity |
| Column customization | Medium scope, low urgency |
| CSV export | Easy point-release add |
| Global hotkey | 3 different platform APIs |
| Localization / i18n | 500+ strings |
| Event log viewer | Large scope, Windows-only |
| Geo-IP lookup | 50 MB embedded DB, privacy concerns |
| Disk Analyzer extras (delete files, compare scans, filter by type) | Post-ship enhancements |
| Process token / security info | Niche, not expected in a system monitor |
| System tray graphs | Animated tray icons are complex |
| Highlight new/changed processes | Visual polish only |
| RPM / Flatpak / Snap packaging | AppImage + .deb covers Linux 1.0 |
| Command-line filtering in rules | Power-user, not blocking |

## Risk Areas

| Risk | Mitigation |
|------|-----------|
| LiveCharts rc4 → stable has breaking API changes | Read full release notes before upgrading; test every chart view |
| Disk Analyzer slow on large Linux filesystems | Test on `/usr` before enabling; add meaningful progress + cancellation |
| Rx test infrastructure setup takes longer than expected | Start with pure-logic services; build momentum before tackling observable mocks |
| Remote Viewer scope expands | Keep it read-only and Prometheus-only; resist adding write operations |
| Snapshot & Compare diff UI complexity | Keep it simple: color-coded rows (green/red/yellow), no animations |
| Quiet Hours × Webhooks × Predictions interaction | Explicit integration tests in phase 25 |
| Screenshot quality requires real data | Run app for 5+ minutes before capturing; Windows-only is acceptable for 1.0 |
