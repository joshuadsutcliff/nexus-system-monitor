# Nexus System Monitor — 1.0 Release Readiness Assessment

> Written: 2026-03-07 · Based on v0.1.7, Phase 18 complete

**Current stats:** 8 projects in solution, 24 AXAML views, 18 navigation tabs, 3-platform provider parity (Windows/macOS/Linux), 12-asset cross-platform release pipeline, 44/51 Process Lasso features (86%), 18/20 WizTree features (90%).

---

## Tier 1: Blockers (Must fix before 1.0)

### 1. Zero Test Coverage

**The single largest risk.** The test project contains exactly one test: `Assert.True(true)`. For an application that manipulates process priorities, kills processes, modifies CPU affinity, and manages system services under admin privileges, this is a safety and credibility concern.

**Minimum viable test suite for 1.0:**
- `RulesEngine` — rule matching, priority application, preference persistence
- `MetricsStore` — rollup calculations, retention cleanup
- `ProcessPreferenceStore` — CRUD operations, thread safety
- `SettingsService` — save/load, migration, corruption recovery
- `AnomalyDetectionService` / `SystemHealthService` — health score calculation, bottleneck detection
- `PerformanceProfileService` — activation/deactivation, process restoration
- Platform provider mocks — verify each `IPlatformCapabilities` implementation returns correct values

**Estimated scope:** ~200-400 unit tests across 15-20 test files.

**Phase target:** Phase 20

### 2. No Structured Logging ✅ FIXED (Phase 19)

~~The app has a crash logger for unhandled exceptions, but **zero runtime logging**.~~

**What was done:**
- Integrated `Serilog` with rolling file sink (`%AppData%\NexusMonitor\logs\nexus-.log`)
- `Microsoft.Extensions.Logging` wired into DI — all services get `ILogger<T>` via constructor injection
- Key services instrumented: `RulesEngine`, `GamingModeService`, `AutoBalanceService`, `PerformanceProfileService`, `MetricsStore`, `SettingsService`
- Startup/shutdown logging via `Serilog.Log.Fatal/Warning`

### 3. No Code Signing

For a system tool that **requests admin/root privileges**, unsigned binaries are a major trust barrier:
- **Windows:** SmartScreen shows "Windows protected your PC" — many users will not click through
- **macOS:** Gatekeeper blocks the app entirely — requires right-click > Open or `xattr -d` workaround
- **Linux:** Less critical, but GPG-signed packages are expected for .deb repos

**What's needed:**
- Windows: EV code signing certificate (~$200-400/year) or standard cert + reputation building
- macOS: Apple Developer ID certificate ($99/year) + notarization via `xcrun notarytool`
- Linux: GPG-sign .deb packages

**Phase target:** Phase 21

Fixed: `Directory.Build.props` now reads `0.1.7`.

## Tier 2: High Priority (Strongly recommended for 1.0)

### 6. Accessibility — Zero Infrastructure

No `AutomationProperties.Name` on any control across 32 AXAML files. No `AccessKey` attributes, no `TabIndex` management, no screen reader support.

**Minimum for 1.0:**
- `AutomationProperties.Name` on all interactive controls (buttons, text fields, toggles, sliders)
- Logical `TabIndex` ordering on key views (Dashboard, Processes, Settings)
- Keyboard accelerators for primary actions (kill process, refresh, search)

**Phase target:** Phase 22

### 7. LiveChartsCore Pre-Release Dependency

`LiveChartsCore.SkiaSharpView.Avalonia 2.0.0-rc4` is a release candidate. LiveCharts 2.0.0 stable has been released — upgrade.

**Phase target:** Phase 21

### 8. Nmap Target Input Sanitization ✅ FIXED (Phase 19)

~~`NmapScannerService.cs` appends the user-provided `Target` string directly to nmap arguments without validation.~~

Fixed: `SanitizeTarget()` validates against an IP/CIDR/hostname regex allowlist before building args.

### 9. Auto-Update Mechanism

No update checking exists. For a persistent system tool, users need at minimum a "new version available" notification. GitHub Releases API makes this straightforward.

**Phase target:** Phase 21

### 10. Disk Analyzer State

The DiskAnalyzer engine is fully built and the sidebar nav item exists, but the README still says it's "currently disabled." For 1.0: either fully enable it with confidence, or remove the nav item.

**Phase target:** Phase 23 (decision point)

## Tier 3: Recommended (Would strengthen a 1.0)

### 11. Onboarding / First-Run Experience

No welcome screen, tutorial, or guided setup. The app launches directly into Dashboard with 18 navigation tabs.

### 12. Settings View Discoverability

SettingsView.axaml is 84.5 KB — the largest view in the app — as a single scrollable column. Add search filtering or tab-based categorization.

### 13. Multi-Select Process Actions

Users coming from Task Manager or Process Lasso expect Ctrl+click selection with bulk kill/priority-set.

### 14. Missing Empty/Loading States

Several data-table views have no loading indicator during initial enumeration and no empty state.

### 15. Help / Documentation Access

No in-app help, no keyboard shortcut reference, no "What's New" display, no "Report Issue" link.

### 16. Community Infrastructure

For a public open-source 1.0: `CONTRIBUTING.md`, GitHub issue templates (bug report, feature request), `.github/FUNDING.yml`.

## Bottom Line

The app has impressive feature breadth, genuinely excellent cross-platform architecture, and a mature release pipeline. The UI is polished well beyond typical v0.x software — 18 theme presets, Crystal Glass effects, thoughtful platform-specific gating, and consistent design language across 18 tabs.

**The four biggest gaps are all infrastructure, not features: tests, logging (now done), code signing, and accessibility.** These are the things that separate a personal project from software the general public trusts with admin privileges on their machine.

Addressing Tiers 1-2 (roughly Phases 20-22) produces a strong 1.0. Tier 3 items would make it a great 1.0 but aren't blockers.
